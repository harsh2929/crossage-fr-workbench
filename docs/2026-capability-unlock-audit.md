# Vintrace — 2026 Capability Unlock Audit

> **Scope.** Next-set use cases Vintrace can unlock for 2026 users — both *as the code stands today* and *with additions* — graded **as-is / hybrid / needs-code**.
> **Method.** Generated from a multi-agent audit: 6 subsystems mapped against the live code (161 capabilities, every claim cited to `file:line`), 7 web-research dossiers on 2026 needs/regulation/verticals, 65 candidate use cases adversarially verified against the repo and the product stance, a completeness critic + gap-fill round (→ 73 verified use cases). Verifier corrections are folded inline; forward-looking legal claims that cannot be checked from the repo are flagged for counsel review (see §10.2).
> **Stance (non-negotiable).** Recall-first, human-in-the-loop · consent-gated · local-first / on-device · governance-native. Nothing here proposes autonomous identification, covert/watchlist use, or commercial deployment on research-only weights.

## 1. Executive Summary

**What Vintrace is.** Vintrace is a cross-age face recognition *desktop workbench* — an Electron + React/TypeScript frontend over a local Python backend (InsightFace ArcFace 512-d embeddings, ONNX Runtime, FAISS, HDBSCAN) — built for one job: letting a trained human *review* machine-surfaced "possible same-person-across-time" candidates and adjudicate each one. It is explicitly **not** an autonomous identification system. The product surface is large and shipping: ~96 backend commands, ~75 MCP tools/resources/prompts, an 8-language (incl. Arabic RTL) UI, and a deep governance subsystem (consent receipts, append-only audit, retention reports, Safe Mode NSFW screening, privacy reports, workspace lock).

**The moat (three structural commitments, each both a differentiator and a legal necessity).**

- **Recall-first, human-in-the-loop.** The matcher surfaces ranked leads in calibrated confidence bands (`confident ≥0.40`, `likely ≥0.28`, `child-bucket maybe ≥0.20`, `below-review`) with risk flags; *every* accept/reject/uncertain is a logged human decision (`scoring.py:33-40`, `enroll/manager.py:2215`, `App.tsx` ReviewView). The UI never asserts identity — it says "Strong/Likely possible match." This posture is mandated by the data: NIST IFPC 2025 shows wide-gap child→adult recognition collapsing (TAR@0.1%FAR 98.5% @2yr → 71.3% @8yr). Wide-gap matching is provably unsolved; the product's honesty about that is the wedge.
- **Consent-gated.** Processing is blocked until a human grants scoped consent (`api_server.py:3987 _require_consent`, enforced before every enroll/scan). MCP agents *structurally cannot* grant consent — granting requires `VINTRACE_MCP_OPERATOR_TOKEN`, which an agent cannot mint (`mcp_server.py:465-491`).
- **Local-first / on-device.** Recognition runs entirely on Apple Silicon (CoreML), NVIDIA (CUDA/TensorRT), or CPU; the recognition path has zero network calls. The only egress points are opt-in model/dataset downloads. No face ever leaves the device.

**The 2026 unlock thesis.** Five forces converge in 2026 to make a governance-native, recall-first, local-first biometric workbench *more* valuable than a higher-accuracy cloud matcher: (1) the EU AI Act high-risk biometric regime (conformity assessment, bias testing, EU-DB registration); (2) the US biometric-litigation wave (BIPA private right of action, Texas CUBI behind Meta's $1.4B settlement, Colorado/Washington laws); (3) a sharp local-first / data-sovereignty turn ("the year of sovereign AI"); (4) age-assurance regulation pushing privacy-preserving estimation, *not* face-matching, as the legal architecture; and (5) agentic/MCP adoption demanding exactly the human-only-consent + audit + redaction controls Vintrace already ships. The opportunity is to **convert the governance architecture from overhead into the sellable product** — and to do so right now, while the substantive controls already exist in code.

**The single biggest opportunity.** A **compliance / governance-evidence pack** (DPIA + FRIA + Annex IV technical documentation + tamper-evident audit, generated from data the workspace already collects) is the highest-leverage 2026 unlock. It is stance-reinforcing (it *documents* the recall-first/human-in-loop/consent posture rather than expanding recognition), it is buyer-unlocking across *every* vertical (LE, NGO, healthcare, insurance, eDiscovery), and it rides a real 18-month procurement-readiness window. It is the one feature that turns "we are governance-native" from a claim into an exportable artifact a DPO can hand to a regulator.

A hard caveat that gates *commercial* go-to-market across the whole portfolio: InsightFace **code** and pretrained weights are MIT, but the **training datasets** (MS1MV2/Glint360K/WebFace) are non-commercial research only. Paid deployment to any commercial/LE/NGO customer requires licensed (or self-trained) weights. This does not affect any *as-is* personal/research capability, but it is a revenue precondition for every paid vertical and is flagged throughout.

## 2. Method & Framing

### 2.1 How use cases are classified

Each use case is graded on whether the *current code* supports it:

| Status | Meaning |
|---|---|
| **As-is** | The workflow is assembled from shipping primitives; no new ML, no new commands required for the core claim. |
| **Hybrid** | Most of the value exists; a bounded set of small-to-medium additions (a new lane, a schema field, a composing command, a UI panel) is required to deliver the headline. |
| **Needs-code** | The headline deliverable genuinely does not exist; net-new code (often a new subsystem or data path) is required. |

Effort is sized **S / M / L / XL**. A recurring verifier lesson is folded in: *"the data exists" is not "the feature exists."* Several use cases that looked as-is were downgraded because the load-bearing piece (a candidate-side capture date, a per-subject consent record, a merge *lane*, a regression *metric*) was absent even though an adjacent primitive existed.

### 2.2 The 2026 forces driving demand

- **EU AI Act high-risk biometric obligations** — conformity assessment, bias/accuracy testing, logging, human oversight (Art 14), EU-DB registration. The product stance anchors applicability at **2 Aug 2026**; some catalog entries cite a Digital-Omnibus deferral to Dec 2027. *These deferral dates are external, forward-looking legal claims that cannot be verified from the repo* — they are presented as the strategist's market thesis and the canonical product-stance date (2 Aug 2026) is used where they conflict. The Art 5(1)(h) missing-persons/victim-search carve-out exempts the *prohibition*, **not** the high-risk regime.
- **GDPR Art 9** special-category processing and **DPIA** obligations — unaffected by any AI Act deferral; apply *now*. DPIA is the universal EU entry ticket.
- **US biometric patchwork** — BIPA ($1K–$5K/person, private right of action), Texas CUBI ($25K/violation, AG-only, the lever behind Meta's $1.4B settlement), Colorado CPA, Washington My Health My Data (private right of action). A live litigation wave (Clearview/Speedway settlements, Disney FR class action, breach litigation against centralized biometric stores).
- **Local-first / data-sovereignty** — regulated buyers demand air-gapped stacks where "no query, response, or metadata leaves," each control mapped to a recognized framework (e.g. NIST 800-53). On-device inference is the 2026 default expectation.
- **Age-assurance laws** — the regulator-blessed architecture is privacy-preserving *estimation/ZKP*, **not** 1:N face-against-gallery matching. Face matching cannot win the age-gate market and must position as QA/bias-testing, never as the gate.
- **Agentic / MCP** — MCP became enterprise-standard middleware in 2026; the consensus mitigations (least-privilege, explicit human approval for sensitive tools, audit-grade logging) are exactly what Vintrace ships natively, for the canonical highest-risk tool class (biometrics).

### 2.3 Hard constraints (non-negotiable)

1. **Commercial-weights licensing gate.** Non-commercial research training data caps every *paid* vertical until licensed weights are sourced. Personal/research/non-commercial use is unaffected.
2. **Child→adult accuracy ceiling.** Wide-gap recognition is provably weak (NIST IFPC 2025). Output must remain a recall-first lead; the "child-bucket maybe" band (0.20–0.28 cosine) must always read as *low-confidence, needs human review*.
3. **Stance lines that must never be crossed:** no autonomous identification, no covert/watchlist use, no live public-space 1:N, no synthetic/age-progression *generation* of faces, no agent-granted consent, no commercial deployment on research weights.

## 3. Unlockable NOW (as-is)

These use cases the shipping code already supports end-to-end. For each: who, value, the exact existing code basis, and positioning. (Verifier caveats are preserved inline.)

### 3.1 Long-term missing-child cross-age review queue
**Who:** NCMEC-style missing-children analysts, forensic artists, LE long-term-missing units, cold-case investigators.
**Value:** A 4-person forensic-artist team (NCMEC's actual 2025 capacity: 187 age progressions) reviews machine-surfaced adult candidates at scale, in NIST-grounded confidence bands with forced human adjudication — the explicit cross-age gap NCMEC names but has not operationalized.
**Code basis:** `enroll/manager.py:532-567` (`enroll_age_groups` — child/adolescent/adult/unknown buckets); `mcp_server.py:508-531`, `:1277-1283` (`enroll_age_reference_set`, `plan_multi_age_enrollment` "consent-first plan"); `scoring.py:33-56` (bands + pose-relaxed thresholds); `enroll/manager.py:1597-1660` (`review_insights` strongest-first triage); `api_server.py:3548-3632` (triage lanes); ReviewView + keyboard/bulk accept-reject (`App.tsx`).
**Positioning & caveats:** State explicitly that the **forensic artist's hand-drawn age-progression image is enrolled as an ordinary reference** — the tool performs **no** age-progression synthesis (grep-confirmed). The "multi-thousand-case backlog at scale" claim is **per-case/sequential**, not a concurrent cross-case dashboard (single active workspace, `workspace_registry.py:28-29`); a multi-workspace switcher (§7.3) is needed for true cross-case throughput. Children's biometrics are GDPR Art 9 heightened special-category; the consent receipt (`api_server.py:4318`) and audit log already support DPIA evidence. Risk: **medium** (highest-sensitivity data class in the most-scrutinized lane). Commercial NGO/LE deployment gated by licensed weights.

### 3.2 Air-gapped CSAM / trafficking victim-ID triage bench with Safe Mode shielding
**Who:** Anti-trafficking NGO analysts (Marinus/Thorn-*style* — analogous workflow, not an integration/endorsement), LE child-exploitation units, CSAM victim-ID examiners.
**Value:** Offline, no-egress victim-ID triage where the AdamCodd ViT NSFW gate screens media before a human sees it, with full audit — addressing both over-reliance on cloud tooling and analyst-wellbeing-at-scale.
**Code basis:** `ingest/safety.py:35-126`, `:235-281` (hybrid heuristic + AdamCodd ViT ONNX); `:311-334` (pinned SHA-256 + integrity verify, fail-closed-to-heuristic); `enroll/manager.py:1050-1079`, `:1416-1448` (Safe Mode protected-phase exclusion from matching/clustering — *inline mechanism, no `_safe_is_protected` function*); `:4436-4492` (`export_safe_mode_audit`); `mcp_server.py:613-629` (`assess_image`, consent-gated, path-confined); `embed/engine.py:136-219` (local ArcFace, no cloud).
**Positioning & caveats:** Safe Mode **reduces, does not eliminate** exposure — `safe_mode_face_crop_allowed` (`manager.py:997-1027`) deliberately admits borderline-sensitive images with a centered face into matching (a recall-vs-protection tradeoff). For the CSAM vertical, expose this carve-out's count prominently and add a **zero-admittance toggle** (this nudges a *deployable* vertical to hybrid). Effort **M** (chain-of-custody case management and tamper-evident retention are *not* implemented — only a generic audit log). Risk: **high** (domain, not code defects). Hard gates: licensed weights for any paid LE/NGO sale; EU AI Act high-risk obligations apply (this is squarely a victim-ID bench).

### 3.3 "Who is this in our old photos?" cross-age person organizer (digitized family archives)
**Who:** Hobbyist genealogists, family historians, people digitizing inherited print/slide archives.
**Value:** Solves the "orphaned photo" problem — groups the same person across decades locally, aging acknowledged as uncertain, human confirms every link, no cloud upload.
**Code basis:** `enroll/manager.py:874-935` (`flush_unmatched` → "Unmatched cluster N" pseudo-people, band `clustered review`, score 0.0); `cluster/clusterer.py:29-57` (HDBSCAN + DBSCAN fallback); `:2550-2581` (`rename_person` casefold merge + audit); `image_io.py:269-299` (capture_date EXIF→mtime fallback, **per image**); `scoring.py:33-40` (graded bands).
**Positioning & caveats:** capture_date is **per photo**, not per face (functionally fine). **Biggest honesty gap:** scanned prints/slides almost never carry original-event EXIF — `capture_date()` falls back to scan-date mtime, so the "across decades" timeline collapses onto the scan date and *misleads exactly this persona*. Mitigate with a user-editable "approximate decade" control + EXIF-vs-inferred provenance display (without it, downgrade the "seen across decades" sub-claim to hybrid). Warn on name-collision merges (Jr./Sr.). Living minors/bystanders are GDPR Art 9 + BIPA/CUBI; keep the "aging is never confirmed identification" disclaimer in the merge UI. Effort **S**, risk **medium**.

## 4. Near-term Unlocks (hybrid — small additions)

Grouped by vertical. Each: value, what exists, required additions, effort, governance/risk.

### 4.1 Missing-persons / humanitarian / DVI

**Consent-gated humanitarian family reunification (ICRC Trace-the-Face pattern).** *Value:* consent-first, human-confirmed, no-government-data-sharing reunification, fully offline in low-connectivity settings, 7-language incl. Arabic RTL. *Exists:* scoped consent + audit (`manager.py:426-470`), MCP operator-token guard, consent-receipt export, multi-age enroll, zero-egress backend. *Required:* (1) **wire consent revocation to purge/quarantine templates** — `set_consent(False)` only flips the active flag and audits; it does **not** purge embeddings (a real GDPR Art 17 erasure gap), though the teardown primitives (`delete_reference`, `clear_references`, `vector_store.clear`) exist; (2) a structured **"freely-given basis"** field (consent note is free-text only) — coerced consent is the dominant humanitarian failure; (3) at-rest workspace encryption for device-seizure scenarios. *Effort:* **M.** *Risk:* **medium.** Reframe value prop to avoid implying it solves wide-gap child→adult accuracy — it is a triage aid for a human, not a matcher.

**NamUs/DVI cross-condition triage to prioritize scarce DNA testing.** *Value:* turns a 15,000+ unidentified-remains backlog into a DNA-testing priority queue; the strongest stance-aligned use case in the set (triage → DNA-confirmed). *Exists:* low-quality/pose rescue (`engine.py:178-258`), fused scoring with risk flags, multi-age ante-mortem enroll, lanes, review ledger. *Required (effort L, two additions heavier than they look):* (1) a **triage-priority export** ranked by `fused_score × age-gap-reliability` — but there is **no per-age-gap calibration signal in code**; an honest multiplier requires a NIST-style TAR-by-gap table or a documented heuristic discount (otherwise it fabricates confidence); (2) a **required (not optional) cross-condition warning** — ArcFace has *no validated accuracy* on post-mortem/3D-reconstruction imagery; this flag must gate the whole mode; (3) a **forensic/legal-authority consent mode** — deceased subjects cannot consent; lawful basis is the investigating authority, so the gate must record case number + authorizing agency + legal basis, not subject permission. *Risk:* **medium.** Bias surfacing + EU AI Act high-risk obligations + licensed weights all apply.

**Forensic-artist productivity multiplier (backlog batching + reference refresh).** *Value:* a tiny team triages a tens-of-thousands-case backlog with batched queues, video-moment grouping, and resumable scans. *Exists:* `video_moments`, `backfill_references_for_model` (model migration), `apply_review_rules`, resumable manifests, saved scan sources/queue. *Required:* (1) reframe "periodic re-enrollment per NIST" — there is **no scheduler/staleness detection**; `backfill` re-embeds existing refs for a *new model*, it does not pull newer photos. Add a per-person **reference-staleness signal** (capture_date age vs interval) + a "references due for refresh" queue; (2) saved scan sources/queue are **frontend-only localStorage** (per-device, capped) — for team-scale backlog, durability must move into the workspace DB; (3) `bulk_set_candidate_status` audits only an aggregate row — for Art 14 add **per-candidate reviewer + timestamp** and consider gating bulk *accept* behind stricter confirmation. *Effort:* **M.** *Risk:* **medium** (rubber-stamp / Art 14).

### 4.2 Genealogy / personal / legacy

**Inherited-archive triage for a deceased parent (non-griefbot).** *Exists:* consent/receipt/retention/privacy machinery, `systemPhotoSources` discovery. *Required:* a structured **subjectStatus (deceased/living)** field on consent (bump `schemaVersion`, migrate, add to JSON **and** CSV writers), i18n preservation-vs-recreation copy (no deceased/legacy strings exist today), a per-person living/deceased tag. ## 6. Vertical Deep-Dives

### 6.1 Missing-persons / humanitarian / child-safety (lead vertical)

This is the strongest fit for the product stance and the densest cluster of as-is + low-effort unlocks. The demand signals are concrete and unmet: NCMEC's 4-artist capacity against tens of thousands of cases (187 age progressions, 32,167 cases assisted in 2025); NCMEC's own stated, unoperationalized goal of "applying age progression to facial images, including facial recognition"; 21.3M CyberTipline reports with child-sex-trafficking +323%; ICRC Trace-the-Face across 40+ countries; Interpol's 34th DVI Conference (63 countries) standing up an FR sub-working group; NamUs's 15,000+ and Othram's Project 525 backlog. EU AI Act Art 5(1)(h) explicitly *permits* missing-persons/abduction-victim search as a biometric lane (while the high-risk regime still applies).

The competitive gap is exactly the Vintrace posture: incumbents are cloud + scraped + autonomous (Clearview/PimEyes) or cloud-bolt-on (Spotlight/Rekognition-backed). Vintrace is the legal/ethical opposite — offline, consent-gated, recall-first, audited, with NSFW shielding for analyst wellbeing.

**Sequencing within the vertical:** ship the as-is review queue (§3.1) and the air-gapped CSAM bench (§3.2) first (with the zero-admittance Safe Mode toggle and chain-of-custody case management as the M-sized hardening); then the consent-revocation-purge + freely-given-basis additions (§4.1) that make reunification a flagship stance-compliant vertical; then the age-gap uncertainty surfacing (§5.4) as the honesty differentiator. ### 6.2 Genealogy / personal / legacy

The 2026 pivot is from names/dates to **photos** as the primary genealogical artifact, against a ~10–12% CAGR market; mass-digitization services (iMemories/ScanMyPhotos) create exactly the unlabeled-archive backlog this product organizes. ### 6.3 Investigative / forensic / legal

Courts require authenticity + integrity + chain-of-custody *simultaneously* (FISWG/ACE-V/Daubert); practitioners specify exactly the artifacts Vintrace already captures (input hashes, tool version/settings, timestamped human decisions, audit trails). The competitive wedge is honesty: cloud eDiscovery FR bolt-ons (HaystackID/eDiscovery-AI Feb-2026 acquisition framing; Reveal) ship cross-age matching with marketing-grade claims and no exposed recall/FAR-by-age-gap data — inviting Daubert exclusion. ### 6.4 Trust / compliance-as-product

This is where the governance architecture becomes the *product*, not overhead — and it is the cross-vertical multiplier. The thesis: GDPR Art 9/DPIA apply *now* (unaffected by any AI Act deferral), the EU AI Act high-risk regime is coming, the US patchwork is litigating, and buyers across *every* vertical want to **procure audit-ready tooling now** rather than scramble at a deadline. ### 7.3 Team / multi-workspace
Strong per-case isolation already exists (per-workspace lock, consent, audit, retention; `set_workspace` resets engine + re-reads consent; per-workspace localStorage UI state; marker-validated active-workspace pointer). The gap is a user-facing **recent-workspaces registry + topbar switcher** (M). **Load-bearing guardrails:** keep `list_workspaces` **desktop-IPC-only or path-redacted** (a new command returning case names+paths over MCP would leak cross-case data, breaking the isolation it sells); give the registry the same **marker-validation** as the active pointer (anti-poisoning); compute lock status at render time; cache last-known pending counts rather than fanning out across inactive DBs; show destination consent state on switch. The on-prem team MCP host (§6.5) adds per-analyst identity, scoped tokens, rate-limiting, and audit-deletion gating.

### 7.4 Video / CCTV (as-is, §3.9)
Temporal person-moments are the difference between a usable and unusable footage workflow and the named SIU/DFIR media-analysis gap. Strictly post-hoc on lawfully-held footage; the only small add is a reviewer-facing decode-failure lane if promised.

### 7.5 Age-assurance module (QA-only, §5.3)
The deliberate *refusal* to be the age gate, repositioned as a NIST-FATE-style estimator QA/bias harness, is itself a strategic enabler — it lets the product engage the largest 2026 regulatory tailwind without entering an unwinnable (and legally wrong) face-matching-as-age-gate market.

### 7.6 Internationalization, accessibility & four-eyes review (procurement enablers)
- **Non-English / RTL deployment (hybrid, M):** seven complete translation tables (en/zh/es/fr/ar-RTL/hi/ja), full RTL flip, translated Electron menu, locale-aware boot HTML. The seam: **backend-generated compliance artifacts (DPIA, consent receipts, ledgers) are English-only** — a non-English regulator may deem an English consent receipt non-compliant. Also reconcile the "8-language" claim with the **7** actually wired, add locale-aware date/number/name-order formatting, and professionally legal-translate consent/disclaimer strings (a mistranslated "investigative lead, not identification" disclaimer weakens the oversight posture). Unlocks the inherently-multilingual government/NGO segments.
- **Accessibility-conformance (Section 508 / EN 301 549) (hybrid, M):** the substantive a11y behaviors ship and are CI-tested (focus-visible, keyboard-operable review with single-key decisions, modal focus trap, forced-colors/reduced-motion, ARIA live regions, RTL). The procurement gate is a **truthful VPAT 2.5 / ACR + EN 301 549 statement** (must mark Partially/Does-Not-Support where H6 localized aria-labels and H7 row-focus are unfixed — an over-claiming VPAT is itself fraud exposure), an `a11y.*` i18n namespace for the 65/69 hardcoded-English aria-labels, an axe-core scan backing the claims, and an accessibility line item in `release_readiness`. Local-first means no inaccessible cloud-login wall to remediate. Accessibility is necessary-not-sufficient — it does not cover the AI Act/GDPR/BIPA/weights gates.
- **Dual-control four-eyes review (needs-code, M):** an enforced reviewer-A-then-different-reviewer-B state machine with a disagreement lane, mapping 1:1 onto ENFSI/FISWG peer review and Art 14. Net governance-positive and the concrete antidote to "reviewer-automation complacency." Required: thread an explicit **reviewerId** through the decision path (do **not** default to session actor for finalizing actions — today no per-decision reviewer identity exists); a `dual_control` config flag (off/accepts-only/cross-age-only/all); ReviewCandidate + schema extension (review_stage, first/second reviewer/decision/rationale, agreement); a `confirm_second_review` that **rejects when reviewerB == reviewerA** (server-side); new lanes + audit actions; an MCP guardrail so an agent can never serve as reviewer A or B. Reviewer identity becomes redactable PII in exports.

## 9. Risks, Guardrails & Out-of-Bounds

**What Vintrace must NEVER do (and why):**

- **Autonomous identification.** Wide-gap child→adult matching is provably unsolved (NIST IFPC 2025 TAR collapse). Any output presented as a positive ID is both a wrongful-ID liability and a Daubert-exclusion/EU-AI-Act-Art-14 failure. The UI must say "possible match"; every decision must be a logged human action. This is enforced in code and must stay so.
- **Covert / watchlist / live public-space 1:N.** Banned under EU AI Act Art 5; the BIPA/Disney litigation magnet. The events/education/camera use cases are in-bounds *only* because they are consent-gated, recall-first, post-hoc, and (for events) 1:few against a consented opt-in roster — ship them only behind hard covert/attendance/watchlist refusal gates.
- **Untargeted scraping.** Art 5(1)(e) "Clearview ban" in force. The anti-scraping guardrail (§5.5) makes refusal a demonstrable control; ingest must never become a vector for bulk un-consented galleries.
- **Synthetic / age-progression *generation* of faces.** No such code exists (grep-confirmed); it would risk the EU Art 5 NCII/CSAM-generation prohibition. The forensic artist's hand-drawn image is enrolled as a reference, never synthesized. "No recreation" of the deceased is a product promise + export boundary.
- **Agent-granted consent.** Structurally impossible (operator token an agent cannot mint). Any new attestation field (FRIA, lawful basis, jurisdiction preset, reviewer identity) must inherit the same human-only gate — never MCP-writable.
- **Commercial deployment on research weights.** MS1MV2/Glint360K/WebFace are non-commercial research only. Every paid vertical is gated on licensed weights; the court-ready report deliberately stamps the research-tier license (a feature, not a bug).

**Standing guardrails:** Safe Mode reduces (not eliminates) exposure — disclose the face-crop carve-out and offer zero-admittance for CSAM. Honesty gates: never render a NIST TAR % as this-match confidence; never imply "local = encrypted/secure at rest" (vectors + previews are unencrypted; rely on OS encryption + Workspace Lock access control); never imply a generated DPIA/VPAT/Annex-IV pack *is* sign-off/certification; never imply air-gap exempts the AI Act high-risk regime; never present a cross-age band, bias-pack run on public datasets, or scraped-gallery proxy count as more than it is. Children's biometrics are heightened Art 9 + BIPA/CUBI throughout.
